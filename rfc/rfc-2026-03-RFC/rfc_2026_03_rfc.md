# Request For Comments 2026-03: RFC Process

* Status: Proposed
* Authors: [Quan Nguyen](mailto:qmn@zerorisc.com), [Robert Schilling](https://github.com/Razer6/)
* Last Updated: April 16, 2026

## Introduction

Large changes to open projects require discussion, collaboration, and iteration beyond the scope of a single pull request or issue.
The Request For Comments (RFC) procedure, inspired by the [IETF RFC](https://www.ietf.org/process/rfcs/) process and similar mechanisms, helps set and justify the technical direction for a project.
Publishing RFCs also provides historical context for past design decisions.
This RFC proposes guidelines for the RFC submission process as well as criteria under which an RFC is suggested or mandatory.

## What requires an RFC?

We expect isolated bug fixes and small enhancements to remain as ordinary pull requests and issues.
The following guidelines outline circumstances in which an RFC is *required* for new contributions.
Nevertheless, code owners may, at their discretion, refer pull requests to the RFC process to request further elaboration of problem context, design decisions, etc. that exceed discussion customarily done in a pull request discussion.

### Introduction of new major IP blocks

The introduction of any new major IP block must be proposed via an RFC.
A simple test for whether a block is "major" is whether it affects other blocks through inter-module connections.
(An example of a non-major IP block is an I/O peripheral whose only connection to the system is through the memory interconnect.)
This ensures proper integration and alignment with the overall architecture.

### Changes affecting security posture

Any change that potentially impacts security posture, whether hardware or software, must be proposed via an RFC.
This includes:

* Introduction of new security features.
* Modifications to existing security features.
* Major changes to cryptographic implementations.

### Architectural changes spanning multiple IP blocks:

Any change that significantly alters the overall architecture of the project or affects multiple IP blocks or subsystems must be proposed via an RFC.
This includes:

* Introduction of new major architectural components.
* Significant modifications to existing architectural components (but see limited exception for feature gating, below).

### Major changes to supporting infrastructure

Any change that adds substantive new functionality to supporting infrastructure (topgen, reggen, ipgen, dtgen, etc.) which is not a straightforward extension of existing features must be proposed by an RFC.
This includes:

* Changes that significantly alter or reorganize HJSON metadata.
* New generators of autogenerated code.
* Substantial templatization or factoring of code into templates.

### Exceptions

Changes that are strictly within the scope of a single IP block and do not impact other parts of the system may not require an RFC, unless requested by a code owner.

Feature additions to existing IP blocks that are behind a parameter (Verilog parameter, ipgen parameter) and do not affect other tops generally do not require an RFC.
This exception applies when the feature is isolated and can be enabled/disabled without impacting the functionality of the IP block or the wider system.
However, if the feature has the potential to impact security, compliance, or the overall architecture, an RFC is still required.

The addition regarding feature gates addresses a common scenario in engineering.
Feature gates (or configuration options) allow for the inclusion of optional features without fundamentally altering the core design.
This addition clarifies that if a feature is well-contained and does not have ripple effects, a full RFC may not be necessary.

#### Why this exception is important:

* Reduces overhead: It prevents unnecessary RFCs for minor enhancements, allowing for faster iteration and development.
* Promotes modularity and flexibility: It encourages the use of configuration options to manage optional functionality, promoting a more modular and adaptable design.
* Encourages iteration: It allows developers to add and test new features without the overhead of a full RFC.

Important Note: Even with feature gates, if a change has security implications, affects compliance, or introduces significant architectural modifications, an RFC is still mandatory.
The goal is to strike a balance between process and efficiency.

## Formatting and submitting an RFC

RFCs are submitted through the pull request process.

1. Create a new subdirectory under `rfc` with a name of the form `rfc-[YYYY]-[##]-[Name-of-RFC]`, where:
* `[YYYY]` is the four-digit year in which the RFC is submitted,
* `[##]` is the number of this RFC (see [Numbering](#numbering), below),
* `[Name-of-RFC]` is a brief, hyphen-separated name of the RFC.
2. Write the contents of the RFC in a single Markdown file (see [Content of the RFC](#content-of-the-rfc), below).
   This file should be titled `rfc_[YYYY]_[##]_[Name_of_RFC].md`, noting the use of underscores rather than hyphens.
   The RFC should contain the following header:
```
# Request For Comments [RFC number]: [Title of your RFC]

* Status: Proposed
* Authors: [Your Name](mailto:your_email@example.com)
* Last Updated: [Month Day, Year]
```
3. Open a pull request to add your RFC, and open any issues or pull requests containing code to support your RFC as needed.
4. Discuss your RFC using comments and the code review feature, just as you would with ordinary code.
5. Edit your RFC at any point as you would an ordinary PR by committing new changes and (force-)pushing your changes to the pull request.
   Refrain from rebasing in a way that erases prose from the commit history, even if it is deleted as a result of discussion.
6. When appropriate, have the RFC brought up for a vote (see [Approving an RFC](#approving-an-rfc), below).
7. When the vote is complete,
   1. Change the RFC header's Status to the result of the vote, "Approved" or "Not Approved".
   2. Update the "Last Updated" date.
   3. Assign the RFC number

### Content of the RFC

There is no specific format for the rest of the RFC.
The RFC should contain enough information to enable focused discussion among technical contributors and decision makers.
It's best to place at the top of your RFC either (1) a summary of the proposed changes or (2) the principal matter your RFC seeks to resolve.
Code snippets may be embedded in the RFC, but proposed changes should remain in pull requests referenced by the RFC.
Code is for illustrative purposes only; no code outside of the rfc/ repository may use any code in the rfc/ directory.

### External media

[Images](https://github.github.com/gfm/#images) can be added to the RFC's subdirectory.
Use relative links to external media.
When possible, use file formats which enable (1) later modification and (2) reproduction at scale, such as vector-based graphics formats like SVG.

### Numbering

Each RFC can be identified by a year and a number.
This RFC is 2026-03.
RFCs are ordered by the age of their corresponding pull request, with older pull requests coming first.
Any newly opened PR with an RFC receives the next number after the most recently proposed RFC, zero-padded to two places.
The first RFC in any given year receives the number 01.
When opening a PR, assign the next available number.
The final RFC number may change before merging.

## Discussing and changing a proposed RFC

The full justification and discussion for any RFC must be publicly readable.
In this project, discussion can occur on the RFC pull request through comments.
Discussions may also take place on the project mailing list.
This is because the RFC is more than just the files in the pull request; the discussion often shapes the RFC and its final approval.
When making changes to the RFC, _do not_ amend or otherwise alter the commit history; instead add subsequent commits.
Making changes by subsequent commits helps reviewers better understand changes to the RFC.
At merge time, squash any subsequent commits into a single commit.

## Approving an RFC

The Technical Steering Committee (TSC) votes on RFCs.
A proposer of the RFC calls for a vote by leaving a comment in the pull request discussion and sending an email to the TSC.
The TSC will then conduct a vote at the next regularly scheduled meeting.

The TSC may choose to approve an RFC, may choose _not_ to approve an RFC, or determine that the RFC should be decided through the ordinary PR review process (in other words, "remanded").
The third option---deciding a PR through ordinary PR review---may be used for RFCs that principally concern technical implementation details or otherwise do not substantively alter the technical direction of the project.

At the completion of a vote, the TSC announces the result of its vote and updates the status of the RFC to "Approved", "Not Approved", or "Decided by PR review process".

*Unlike pull requests with code changes*, RFCs with substantial discussion (and not otherwise closed for being trivial, a duplicate, or not meeting the structural standards for RFCs) should be merged regardless of whether the RFC is approved.
Merging all RFCs preserves discussions and technical decisions for future reference.

RFCs decided through the ordinary pull request process should be followed up by a separate pull request making the changes described by the RFC.

## Storing and presenting RFCs

RFCs, external supporting media, and digests of conversations (outside of pull requests) will be stored in the rfc/ directory as described above.
While the RFCs are human-readable from the rfc/ directory, this organization is intended to support collating and compiling RFCs into a book presented as a webpage such as the Rust RFC book.

## Obsolete and superseded RFCs

An RFC, once merged, should no longer receive substantial updates other than clerical or administrative fixes.
Clerical changes include obvious typographical errors in prose; administrative fixes include file renames that preserve the integrity of the RFC.
Instead, existing RFCs are superseded or obsoleted by new RFCs.
